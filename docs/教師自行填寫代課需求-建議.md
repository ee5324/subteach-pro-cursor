# 請假老師自行填寫代課需求 — 建議做法

## 原則：中繼站 ＋ 管理員審核

**請假老師填寫的代課需求，一律先進入「中繼站」（待審區），不直接寫入主系統。**  
只有管理員在後台**審核並執行「匯入」**後，該筆申請才會變成正式的請假紀錄（LeaveRecord）並出現在待聘清單／代課總表。這樣可以：

- 避免誤填、重複或不合規的資料直接進系統
- 由管理員確認假別、節次、支薪方式等再放行
- 可選擇「不匯入」或「歸檔」不通過的申請

---

## 流程概覽

```
請假老師填表  →  中繼站（待審區）  →  管理員審核  →  匯入  →  主系統（請假紀錄／待聘清單）
                     ↑                      ↑
               僅存放待審資料           僅登入管理員可讀／匯入／歸檔
               不進主系統
```

---

## 現況（已是中繼站設計）

- **GAS 教師請假申請**（`gas/TeacherRequest.html`）：老師填寫後送出，寫入 Google 試算表「**請假申請候審區**」（即中繼站），**不會**自動寫入主系統。
- **待審申請**（`#/requests`）：管理員登入後在此頁「從 GAS 載入」看到中繼站資料，逐筆審核後點「**匯入**」才建立請假紀錄；亦可「隱藏／歸檔」不匯入。
- 匯入後的流程不變：至**待聘清單**設公開 → **發佈公開** → 公開缺額公告顯示。

---

## 建議方向（擇一或並行）

### 方案一：沿用 GAS 表單（改動最少）

- **中繼站**：Google 試算表「請假申請候審區」——老師送出後**只寫入這裡**，不會自動進主系統。
- **老師**：開啟 GAS 表單（例如 `?page=request` 的網址），填寫請假與代課節次後送出。
- **管理員**：登入系統 → **待審申請** → 從 GAS 載入（讀取中繼站）→ **手動審核**後點「匯入」才建立請假紀錄；不匯入可歸檔。之後至 **待聘清單** 設公開 → **發佈公開**。
- **優點**：不需改程式，表單已支援完整代課節次，且本來就是中繼站＋審核設計。
- **缺點**：依賴 GAS 與試算表；老師與管理員要記兩套入口（系統 vs GAS）。

---

### 方案二：新增「教師請假／代課需求」Firebase 表單（推薦）

- **中繼站**：Firestore 專用 collection `teacherLeaveRequests`。老師送出的資料**只寫入這裡**，不會寫入 `records` 或任何主系統 collections。
- **老師端**：對外表單頁（例如 `#/teacher-request`，不需登入）。填寫姓名、假別、事由、請假區間、支薪方式、代課節次明細後送出 → 僅寫入 `teacherLeaveRequests`，狀態 `pending`。
- **管理端**：登入後在 **待審申請**（或「教師請假待審」）讀取 `teacherLeaveRequests` 中待審項目，**手動審核**後：
  - **匯入**：建立教師（若無）、建立一筆 `LeaveRecord`、節次為待聘（或自覓代課人），並將該筆中繼站文件標記為已處理（例如 `status: 'imported'` 或移出待審列表）。
  - **不匯入／歸檔**：僅標記為已審、不匯入（例如 `status: 'rejected'` 或歸檔），主系統完全不新增資料。
- **後續**：與現在相同，匯入後到 **待聘清單** 設公開並 **發佈公開**，公開缺額公告才會顯示。

**優點**：

- 老師在同一個網站填表，不必開 GAS；資料全在 Firebase。
- **一定經過中繼站與管理員審核**，不會直接進入主系統。
- 與現有「待聘清單 → 發佈公開」流程相容。

**實作要點**：

1. 新增表單頁 `TeacherLeaveRequest.tsx`，路由 `#/teacher-request`；送出時只寫入 `teacherLeaveRequests`（`status: 'pending'`）。
2. Firestore 規則：`teacherLeaveRequests` 允許未登入 `create`；讀取／更新／刪除僅限已登入（管理員）。
3. 待審申請頁：新增「從 Firebase 載入」或分頁，僅顯示 `status === 'pending'`；匯入時建立 LeaveRecord 並更新該筆為已處理，不匯入時僅更新狀態或歸檔。

---

### 方案三：GAS 與 Firebase 表單並存

- 保留現有 GAS 教師請假表單給已習慣的學校使用。
- 同時提供方案二的 Firebase 表單，管理員在待審申請可選擇「從 GAS 載入」或「從 Firebase 載入」。
- 兩邊匯入後都進同一套請假紀錄與待聘清單，發佈流程一致。

---

## 建議結論

- **短期**：若已在使用 GAS 表單，可先**沿用方案一**，把「教師請假申請」連結與「待審申請 → 匯入 → 待聘清單 → 發佈公開」流程對老師與行政說明清楚。
- **中長期**：建議**採用方案二**，在系統內新增「教師請假／代課需求」Firebase 表單，讓請假老師在同一個網站自己填寫代課需求，管理員只負責審核匯入與發佈，資料與流程都集中在 Firebase 與本系統。

若你決定做方案二，可再指定要我先做「表單頁＋Firestore 寫入」或「待審申請讀取 Firebase 並匯入」其中一側，我可以依現有結構寫出具體程式修改步驟或程式碼草稿。
